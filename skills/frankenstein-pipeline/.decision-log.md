# Decision Log — frankenstein-pipeline

## Session 1 — 2026-05-14

### Classification
- **Complex Workflow** — six stages, two conditional feedback loops, two human checkpoints. Stages carved to `references/`.
- Standalone (not module-based).
- No headless support — human checkpoints are core to the design.

### Spec Decisions
- **Fixed configuration** — no customize.toml for hackathon scope. Natural configurability surface for later: model routing per agent, iteration caps, Docker timeout, tool library path, Chroma collection names.
- **Checkpoints inline** in SKILL.md — short gates, not complex stages.
- **Loop caps tracked in workflow** — MAX_SPEC_ITERATIONS=3, MAX_BUILD_ITERATIONS=3, MAX_ELICITOR_ROUNDS=3.
- **Learner produces build report** — BuildOutcome is the post-build record.
- **Depth: very deep** — each reference includes full Pydantic models, prompt engineering notes, example I/O (PS-08 Loan Underwriting), error handling, edge cases.

### Design Rationale
- **Cross-model adversarial review** — Architect (Claude) and Critic (GPT-4o) use different model families. Same-model review shares blind spots.
- **Template-driven code gen** — Builder compiles from templates, not free-form. Deterministic > creative for code generation.
- **Two-level failure routing** — Tester distinguishes code-level (Builder fix) from spec-level (Architect redesign). Prevents unnecessary spec churn for simple bugs.
- **Model-per-agent via OpenRouter** — each agent gets the model optimized for its task. Elicitor/Tester/Learner use cheap fast models; Architect/Builder use strong reasoning models.

### Alternatives Considered
- **Single model for all agents** — rejected. Cross-model adversarial review is a core architectural differentiator.
- **Customizable from day one** — rejected. Hackathon scope. Fixed is faster to build, easier to debug.
- **Headless mode** — rejected. Two human checkpoints are the product's trust mechanism. Removing them defeats the purpose.
